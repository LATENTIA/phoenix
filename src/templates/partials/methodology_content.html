Methodology

A complete description of how Phoenix processes every IBKR statement and arrives at the numbers shown in each report. Last updated {{ now }}.
Phoenix is an audit-trail tool, not a tax-advice tool. Every figure is traceable to a row in data.db and back to a line in the original IBKR Activity Statement (CSV) or Flex Web Service (XML) file. Use this page as a checklist when reviewing the output with your accountant.
1. Ingestion pipeline 2. Trades and FIFO lots 3. Corporate actions 4. Transfers between sub-accounts 5. Open positions reconciliation 6. FX conversion (ECB) 7. TOB (Tax on Bourse Operations) 8. Profit and loss 9. Performance metrics 10. CGT 2026+ for individuals (basis reset) 10b. Corporate income tax (CIT) for companies 11. Dividends and foreign WHT 12. Capital events excluded from dividends 13. Deduplication and re-ingest safety 14. Auditing a single number end-to-end

1. Ingestion pipeline

Two file formats are supported: IBKR Activity Statement CSVs (one file per year, manually downloaded) and IBKR Flex Web Service XMLs (current year, auto-fetched via the user's Flex token + Query ID). Both feed into the same SQLite tables, so the report layer is source-agnostic.

Pipeline order

  1. Download: downloaded/<account>/ already contains historical CSVs. The current-year XML is fetched on Load data via core/flex_client.py.
  2. Ingest: ingest.py walks every file, calls the matching loader in core/loaders.py, then writes rows to data.db with a UNIQUE constraint on (source_id, identifying-tuple) so re-ingesting is idempotent.
  3. FX sync: ECB EUR/USD daily reference rates from downloaded/ecb_rates.csv are loaded into fx_rates.
  4. Reports read directly from data.db when the user clicks a tab. No intermediate static files.

Each ingested file appears as one row in source_files with its kind (csv / xml), period, and ingestion timestamp. Dropping that row cascades and removes every dependent fact, so a re-ingest with --force rebuilds cleanly.

2. Trades and FIFO lots

Trade rows are normalised to a single schema (date, symbol, quantity, price, proceeds, commission). XML uses execution-level rows (one per fill); CSV uses order-level rows (already aggregated by IBKR). Either way, the report layer treats them as opaque "events" against a symbol.

Lot accounting

P&L and CGT use first-in-first-out lot matching. Each buy opens a lot with cost basis = price × quantity + commission; each sell closes the oldest open lot first, realising

realised_pnl = (sell_price - lot_cost_per_share) × matched_qty - commissions

Symbol canonicalisation strips IBKR's suffix variants (.OLD, .NEW, exchange tags) so a position that gets renamed by a corporate action keeps its lot history. See reports/_helpers.py::canon_symbol.

3. Corporate actions

Corporate-action rows arrive as free-text descriptions in the CSV. core/loaders.py::_parse_ca_row classifies each row using regex into one of four types:

  • Stock split (forward or reverse): existing lots are rescaled by ratio_new / ratio_old. Cost basis per share is divided by the same ratio so total cost is preserved. Fractional residue becomes a small cash payment, captured as a separate trade row.
  • Cash merger: all open lots are closed at the per-share cash consideration. Realised gain/loss flows into P&L.
  • Stock merger (acquisition): old symbol is closed and a new symbol is opened with the same total cost basis, scaled by the new/old ratio. Lot acquisition dates are preserved (this matters for CGT).
  • Delisting: shares written off at zero. The full cost basis is realised as a loss.

Anything that doesn't match one of the four patterns is logged as type='other' and flagged for manual review.

4. Transfers between sub-accounts

The Transfers section of the Activity Statement records share movements between IBKR sub-accounts (e.g. moving a position from a closed U4 account to a new U7 account). These are NOT taxable events. Phoenix preserves the original lot acquisition date through the transfer so the CGT engine still sees lots as pre-2026 even when they appear in a 2025+ file.

Direction (IN / OUT) is taken verbatim from the statement. Market value is captured for sanity-check; cost basis comes from the originating lot, not the transfer's market value.

5. Open positions reconciliation

Each Activity Statement carries an Open Positions snapshot at its end date. After ingest, Phoenix compares its computed open quantity per symbol against this snapshot. A mismatch indicates a missed corporate action, an unrecognised transfer direction, or a manual journal entry IBKR didn't disclose. These show up in the dashboard's OP snapshots counter and feed the Performance diagnostics tab.

6. FX conversion (ECB)

Every USD figure that needs a EUR equivalent uses the ECB EUR/USD daily reference rate on the relevant date (trade date for trades, payment date for dividends). The rate file is downloaded/ecb_rates.csv and is loaded into the fx_rates table. If the requested date is a weekend or holiday, the most recent prior business-day rate is used (a documented ECB convention). See core/ecb_fx_parser.py::get_eur_usd_rate_for_day.

eur_amount = usd_amount / eur_usd_rate_on_date

This date-accurate FX is what makes the dividend totals match what your Belgian bank statement would show, rather than what a year-end FX would.

7. TOB (Tax on Bourse Operations)

Belgium levies a transaction tax on every secondary-market trade, paid in EUR. Phoenix applies the Belgian TOB rate per asset class (currently 0.35% for stocks, with caps) to each trade's EUR-converted notional. The result is the audit table you submit to your accountant for the annual TOB declaration.

Source: reports/tob.py.

8. Profit and loss

P&L breaks into three pieces:

  • Realised: proceeds minus cost basis on closed lots, summed by symbol and by year.
  • Unrealised: current market value minus cost basis on still-open lots. Marks come from the latest year-end snapshot or live Yahoo close.
  • Total: realised + unrealised. Useful for performance, NOT for tax filings (Belgian tax is generally on realised events only).

9. Performance metrics

The Performance tab reports time-weighted return per symbol and per portfolio, plus per-year net flows. Returns are computed in the trade currency (USD for IBKR), not in EUR, to isolate stock performance from FX drift. A separate row shows the EUR-equivalent return so you can see what you actually earned in spending power.

10. CGT 2026+ for individuals (basis reset on 2025-12-31)

Scope: PERSONAL accounts only. This regime applies to individuals subject to Belgian personal income tax (impôt des personnes physiques). Companies (BV/SRL/NV/SA) use the corporate regime described in section 10b instead. The CGT tab is hidden from business-type accounts in the dashboard.

Belgium introduces a capital-gains tax on individual investors starting 2026-01-01. The rule, as currently drafted: gains on lots acquired before 2026 are exempt up to the share's market value on 2025-12-31. Only the gain accumulated AFTER that reset date is taxable.

Phoenix implements this as:

  1. Fetch 2025-12-31 closing prices for every held symbol from Yahoo (the Refresh year-end marks button).
  2. For lots opened on or before 2025-12-31, replace the original cost basis with the year-end price. The original cost is still kept for the regular P&L view.
  3. Realisations from 2026-01-01 onward use the reset basis. A negative delta (the lot is below its 2025-12-31 mark when sold) becomes a deductible loss under the new rules.

This is a draft implementation. The final law text may shift before 2026-12-31. The CGT tab clearly labels each row as "pre-reset" or "post-reset" so the figure is rebuildable if the rules change.

10b. Corporate income tax (CIT) for companies

Scope: BUSINESS accounts only. This regime applies to Belgian companies (BV/SRL/NV/SA) subject to impôt des sociétés / vennootschapsbelasting. Individuals use the CGT 2026+ regime instead. The Corporate Tax tab is shown only for business-type accounts in the dashboard.

A Belgian company that holds an IBKR portfolio is taxed on its investment activity as ordinary corporate income at the standard {{ cit_rate_pct }}% CIT rate (2025/2026). Phoenix models this in the Corporate Tax tab; the rules are:

What feeds the CIT base

Income typeTreatmentRate
Realised capital gain on stock Ordinary corporate income {{ cit_rate_pct }}%
Realised capital loss on stock Fully deductible against any corporate income reduces base
Realised gain on crypto Ordinary corporate income (no special crypto regime for companies) {{ cit_rate_pct }}%
Dividends received Enter the base at gross amount (before WHT) {{ cit_rate_pct }}%
Foreign WHT on dividends Credited (FTC), not deducted. Capped per dividend at {{ cit_rate_pct }}% of the gross of that dividend. credit
Bond / cash interest Ordinary corporate income {{ cit_rate_pct }}%
Unrealised gains / mark-to-market Not taxed. Belgian companies use historical-cost accounting; impairment losses may apply but Phoenix does not model them. 0%

What is INTENTIONALLY NOT applied

Belgian CIT has a participation exemption (100% exempt dividends and 100% exempt capital gains on shares) that drops the rate from {{ cit_rate_pct }}% to 0% when ALL of the following hold:

  • Participation: ≥10% of the issuer's capital OR ≥EUR 2.5M acquisition value (from tax year 2026, the EUR 2.5M-only path also requires that the shares qualify as financial fixed assets, unless the investor is a small company).
  • Holding period: ≥1 year (or commitment to hold for 1 year forward).
  • Subject-to-tax: the issuer must be properly taxed at the corporate level (anti-tax-haven rule, with seven specific exclusions).

A typical IBKR portfolio of small positions never qualifies on the participation threshold (each line is < €2.5M acquisition cost and well under a 10% stake). The Corporate Tax tab therefore assumes none of the holdings qualify, which is the conservative defensible default. If you do hold a strategic stake that meets the conditions, flag it with your accountant: the tab will over-state the CIT owed on that specific line.

Indicative CIT computation per year

CIT base = realised_net_eur + dividends_gross_eur
CIT due = max(0, CIT base) × {{ cit_rate_pct }}%
FTC credit = min(|foreign_wht_eur|, dividends_gross_eur × {{ cit_rate_pct }}%)
Net CIT payable = max(0, CIT due − FTC credit)

"Indicative" matters. Phoenix can only see what flows through the IBKR statement. The company's tax return adds operating profits, salaries, depreciations, the notional-interest deduction, prior-year losses carried forward, group-taxation rules, and the possible SME reduced rate (20% on the first EUR 100k of profit when several conditions are met, which Phoenix doesn't apply). The number shown in the Corporate Tax tab is an audit-trail input for your accountant, not the final tax bill.

Source

PwC Worldwide Tax Summaries, Belgium / Corporate / Income determination. The 2026 amendment to the participation-exemption requirements is documented there as well.

10c. Reynders tax (bond-fund capital-gains withholding)

Scope: personal AND corporate accounts. Applies to any Belgian investor selling shares of a fund whose asset allocation is more than 10% in debt instruments (bonds, cash-equivalents, money-market). Not applied automatically by IBKR (foreign broker) — the taxpayer declares it.

The Reynders tax (named after former finance minister Didier Reynders, introduced in 2013 and expanded in 2018) is a 30% withholding on the interest-derived portion of capital gains realised on redemption or sale of shares in a bond-heavy fund. As of 1 January 2018 the threshold dropped from >25% bonds to >10% bonds, catching most mixed and short-term funds.

How the interest component is computed

Belgian tax law defines the taxable interest component as:

taxable_interest = accrued_bond_interest_at_sale − accrued_bond_interest_at_buy

For funds without a compliant asset-detail statement ("TIS-compliant"), the tax is computed on the entire capital gain as a punitive default. For TIS-compliant funds, the fund provides the interest breakdown and only the interest slice is taxed at 30%.

What Phoenix does — and does NOT do

  • Phoenix does NOT auto-detect Reynders-eligible funds. IBKR's broker feed does not include the fund's bond-allocation ratio. Detecting it would require pulling the fund's KID / TIS statement separately, which is out of scope.
  • Any ETF / UCITS / SICAV realisation appears in the Belgian CGT tab (individuals) and Corporate Tax tab (companies) as a normal capital gain. That is correct for pure-equity funds. For bond-heavy funds, you must apply the 30% Reynders tax on the interest slice separately in your return.
  • The Dividends tab already flags fund distributions as "not eligible for the €833 exemption", which is a related-but-different rule (see section 11 below).

Practical rule of thumb: if the sold instrument is a bond ETF (AGG, BND, TLT, BNDX, EMB, …), a mixed-asset fund, or a target-date fund with >10% bond allocation, ask your accountant to apply the 30% Reynders on the interest slice of that specific lot's gain before filing.

Source

Belgian Income Tax Code (CIR/WIB92) art. 19bis; law of 26 December 2015 and law of 25 December 2017 (threshold change 25% → 10%). SPF Finances circular 2018/C/48.

10d. Crypto — personal (individual) treatment

Scope: PERSONAL accounts only. Crypto lots are auto-flagged in the CGT tab and shown in a dedicated section (they are NOT taxed at the 10% 2026 CGT rate). For business accounts, crypto is ordinary corporate income at {{ cit_rate_pct }}% — see section 10b.

The 2026 CGT law's "financial instruments" scope covers securities, cash, bonds, funds, and derivatives. Cryptocurrencies are excluded from that definition. For Belgian individuals, crypto gains fall into one of three tax regimes based on how you traded them — the classification is a facts-and-circumstances test the tax administration applies case-by-case:

RegimeRateTypical profile
Normal management of private assets
"Bon père de famille"		 0% Buy-and-hold, no leverage, few trades, no borrowed funds. Still reportable as informational income.
Diverse income
Revenus divers, art. 90, 1°		 33% Speculative activity — frequent trades, use of leverage, active portfolio management outside "normal" behaviour. Flat 33%.
Professional income
Revenus professionnels		 progressive Trading as your main or significant secondary activity. Taxed at the progressive personal-income brackets (up to 50%) plus social contributions.

What Phoenix does

  • Trade ingestion tags each lot as asset_class = crypto based on IBKR's assetCategory field or on the manual-trade form dropdown. This flag flows through the FIFO matcher to the closed-lot rows.
  • The CGT tab excludes crypto lots from the 10% calculation entirely (they are flagged n/a (crypto — excluded from 2026 CGT)) and surfaces the lifetime crypto net + per-year rows in a dedicated banner so the taxpayer can pick a regime with their accountant.
  • The P&L tab continues to include crypto lots in realised P&L — that view is about performance, not tax.
  • Crypto lots are never reconciled against IBKR position snapshots (IBKR statements don't contain crypto positions) — the closed-lot list is the source of truth for realised crypto activity.

Bottom line: Phoenix cannot classify your crypto activity for you (the tax administration applies a fact-based test). What it does is give you the numbers each of the three regimes would compute against, so you and your accountant can make the call and file the correct treatment.

Source

Belgian CIR/WIB92 art. 90, 1° (diverse income); SPF Finances rulings on crypto (SDA / DVB, various); parliamentary Q&A on the 2026 CGT scope excluding crypto assets.

11. Dividends and foreign withholding tax

Dividends are read from two places, in this priority order:

  1. The Dividends and Withholding Tax sections of the Activity Statement CSV (one section pair per year).
  2. The CashTransactions element of the Flex XML (only present if the Flex Query has the Cash Transactions section enabled for both Dividends and Withholding Tax).

Each dividend row is described by IBKR with a free-text string like NKE(US6541061031) Cash Dividend USD 0.37 per Share (Ordinary Dividend). core/loaders.py::_parse_dividend_description extracts symbol, ISIN, currency, per-share amount, type tag, and (for WHT rows) the source-country code.

Belgian tax treatment (foreign broker)

  • Foreign WHT: {{ "%.0f"|format(us_treaty_rate_pct) }}% on US dividends when W-8BEN is filed (treaty rate). {{ "%.0f"|format(us_default_rate_pct) }}% if no W-8BEN was filed (the 15% difference can be reclaimed from the IRS). For other source countries Phoenix records what IBKR actually withheld and does NOT attempt to recompute the treaty rate.
  • Belgian "précompte mobilier": {{ "%.0f"|format(belgian_rate_pct) }}% on the dividend net of foreign WHT. IBKR Ireland is a foreign broker so this is NOT withheld at source; the taxpayer self-declares and pays via their tax return.
  • Annual exemption: the first €833 of ordinary dividends per taxpayer per year is exempt for income years 2025 and 2026 (confirmed by SPF Finances). For foreign dividends the mechanism is exclusion from declaration: the first €833 isn't declared, and the corresponding foreign WHT also drops out.

12. Capital events excluded from dividends

IBKR uses the same "Cash Dividend" wrapper for several payouts that are NOT dividend income. Phoenix detects them by the trailing tag in the description (e.g. (InterimLiquidation)) and drops them from the dividend totals. They belong to the cost-basis / CGT side of the engine, where they're already captured as a realised gain or loss on the underlying trade.

Excluded tags (case-insensitive):

{% for t in non_dividend_types %}{{ t }}{% endfor %}

The match runs against both the parsed type tag and the raw description, so unusually-formatted rows are still caught. A matching withholding-tax row (if any) is excluded too, so the pair vanishes cleanly and doesn't distort the WHT total.

Implementation: core/loaders.py::NON_DIVIDEND_TYPES + _is_non_dividend_payout. The filter is applied in all four loaders: load_dividends_csv, load_dividends_xml, load_withholding_csv, load_withholding_xml.

Real-world example: RGTPQ (a delisted issuer in Chapter 11) paid out $1,928 in 2026 tagged InterimLiquidation. That cash partially offsets the cost basis of the now-worthless shares, so it shows up as a smaller realised loss in the CGT tab, not as $1,928 of taxable dividend income.

13. Deduplication and re-ingest safety

Every fact table has a UNIQUE index that pairs the originating source_id with a row-identifying tuple. Re-running python ingest.py --force is therefore idempotent: it truncates every table, drops every source_files row, then replays the ingest. After a code change to a loader (like adding the capital-events filter), --force is the correct way to rebuild the affected tables.

Without --force, ingestion is incremental: only new files (those whose mtime is later than the recorded ingest timestamp) are re-processed.

14. Auditing a single number end-to-end

Every figure in every report is traceable back to a row in the source file:

  1. Click the value in the report (most tables expose the underlying rows).
  2. Note the source column. It carries the originating filename and row type (e.g. csv-div:U7555888_20250101_20251231.csv).
  3. Open that file in downloaded/ and find the row by its date + symbol + amount.
  4. The math from raw row to displayed figure is described in the corresponding section above.

If a figure doesn't reconcile, the most common causes are: a corporate action that wasn't classified (look in the dashboard log for type='other'), an FX rate missing for a holiday-adjacent date, or a Flex Query that doesn't include the Cash Transactions section (Dividends + Withholding Tax). The Sources tab on the Dividends report lists every ingested file and what it contributed.
